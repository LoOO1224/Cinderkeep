Cinderkeep - Data Driven Edit Guide

Purpose
- 이 문서는 JSON 수치를 바꿀 때 어떤 스크립트에 영향을 주는지 확인하기 위한 팀 가이드입니다.
- Static Data는 Assets/Resources/Cinderkeep/data 아래 JSON에 둡니다.
- JSON은 GameDataManager가 로드하고, 각 컴포넌트는 필요한 값만 가져와 사용합니다.
- 데이터가 없거나 ID가 맞지 않으면 Inspector fallback 값으로 안전하게 동작해야 합니다.

Basic Rule
- JSON 필드명은 C# 멤버변수명과 맞춥니다.
- 예: _id, _resourceId, _gatherAmount, _requiredToolType
- _id는 다른 스크립트나 프리팹이 참조할 수 있으므로 함부로 바꾸지 않습니다.
- 기능 테스트 전에는 한 번에 여러 JSON을 크게 바꾸지 않습니다.
- 수치를 바꿨다면 반드시 어떤 플레이 체감이 바뀌는지 적습니다.

Current Active Link - 4.39 기준
1) harvest_nodes.json
- 위치: Assets/Resources/Cinderkeep/data/harvest_nodes.json
- 로드: GameDataManager.LoadHarvestNodeData()
- 조회: GameDataManager.GetHarvestNode()
- 사용: ResourceNode
- 조절 가능:
  _resourceId: 지급할 자원 종류
  _requiredToolType: 필요한 도구 타입
  _requiredToolTier: 필요한 도구 티어 후보
  _gatherAmount: 기본 채집량
  _maxGatherCount: 추후 여러 번 채집용 후보
  _respawnSeconds: 추후 리스폰 시간 후보
- 현재 기준:
  ResourceNode는 _resourceId, _requiredToolType, _gatherAmount를 우선 사용합니다.
  tools.json의 자원별 배수와 함께 최종 채집량이 결정됩니다.
  데이터가 없으면 ResourceNode Inspector 값을 fallback으로 사용합니다.

2) tools.json
- 위치: Assets/Resources/Cinderkeep/data/tools.json
- 로드: GameDataManager.LoadToolData()
- 조회: GameDataManager.GetTool()
- 사용:
  PlayerToolController: 현재 장착 도구 ID와 도구 타입 확인
  PlayerToolUse: 도구 사용 거리, 범위, 쿨타임 참조
  ResourceNode: 자원별 채집 배수 참조
- 조절 가능:
  _toolType: Axe, Pickaxe, None
  _tier: 도구 티어
  _attackDistance: 채집 판정 거리
  _attackRadius: 채집 판정 반경
  _attackInterval: 기본 채집 쿨타임
  _woodGatherMultiplier: 나무 채집 배수
  _stoneGatherMultiplier: 돌 채집 배수
  _ironGatherMultiplier: 철 채집 배수
  _goldGatherMultiplier: 금 채집 배수
  _adamantiumGatherMultiplier: 아다만티움 채집 배수
- 현재 기준:
  채집 배수는 채집량과 채집 속도에 같이 영향을 줍니다.
  예: 0.5는 절반 효율, 0.25는 매우 낮은 효율, 2는 높은 효율입니다.
  도구 타입이 맞지 않으면 배수가 높아도 채집하지 않습니다.
  도구 데이터가 없으면 PlayerToolUse Inspector 값을 fallback으로 사용합니다.

3) weapons.json
- 위치: Assets/Resources/Cinderkeep/data/weapons.json
- 로드: GameDataManager.LoadWeaponData()
- 조회: GameDataManager.GetWeapon()
- 사용:
  PlayerAttack: 공격력, 공격 거리, 공격 반경, 공격 쿨타임
- 조절 가능:
  _damage: 공격력
  _attackDistance: 공격 거리
  _attackRadius: 공격 판정 반경
  _attackInterval: 공격 쿨타임
- 현재 기준:
  PlayerAttack의 기본 무기 ID는 stone_sword입니다.
  장비 시스템이 붙으면 PlayerAttack.SetWeaponDataId()로 현재 무기 ID를 넘깁니다.
  무기 데이터가 없으면 PlayerAttack Inspector 값을 fallback으로 사용합니다.

4) enemies.json
- 위치: Assets/Resources/Cinderkeep/data/enemies.json
- 로드: GameDataManager.LoadEnemyData()
- 조회: GameDataManager.GetEnemy()
- 사용:
  EnemyLoopConnector: 생성된 적에게 EnemyData 연결
  EnemyStatus: 체력
  EnemyMovement: 이동 속도, 정지 거리
  EnemyAttack: 공격력, 공격 간격
  EnemyDetector: 감지 거리
  EnemyBrain: CinderHeart와 플레이어 타깃 판단 보조
- 조절 가능:
  _health
  _moveSpeed
  _stopDistance
  _attackDamage
  _attackInterval
  _visualScale
  _detectorDistance
- 현재 기준:
  적 체력 기준은 EnemyStatus입니다.
  Damageable은 피해를 EnemyStatus, PlayerStatus, BuildingHp, CinderHeart로 전달하는 통로입니다.

5) enemy_spawn_rules.json
- 위치: Assets/Resources/Cinderkeep/data/enemy_spawn_rules.json
- 로드: GameDataManager.LoadEnemySpawnRuleData()
- 조회: GameDataManager.EnemySpawnRuleDataList
- 사용:
  EnemySpawnPoint: 낮/밤/보스 스폰 주기, 웨이브 수, 최대 생존 수, 적 ID
  GameFlowEnemySpawnDirector: 현재 페이즈와 일차에 맞는 EnemySpawnPoint 활성화
- 조절 가능:
  _day
  _phase: Day, Night, Boss
  _enemyId
  _spawnInterval
  _minSpawnCount
  _maxSpawnCount
  _maxAliveCount
  _spawnWeight
- 현재 기준:
  EnemySpawnPoint는 현재 일차와 페이즈가 맞는 규칙 중 weight를 사용해 후보를 고릅니다.
  데이터가 없으면 EnemySpawnPoint Inspector의 fallback 규칙을 사용합니다.

6) game_flow_phases.json
- 위치: Assets/Resources/Cinderkeep/data/game_flow_phases.json
- 로드: GameDataManager.LoadGameFlowPhaseData()
- 조회: GameDataManager.GameFlowPhaseDataList
- 사용:
  GameFlowController: 낮, 밤, 아침 보상, 보스 접근 시간
  GameFlowTimerHUD: GameRunModel을 통해 현재 페이즈와 시간을 표시
- 조절 가능:
  _day
  _phase: Day, Night, MorningReward, BossApproach, BossFight
  _durationSeconds
- 현재 기준:
  GameFlowController는 _day와 _phase가 맞는 데이터를 찾아 시간표로 사용합니다.
  데이터가 없거나 시간이 0 이하이면 GameFlowSettings Inspector fallback 값을 사용합니다.

Prepared Schema - 아직 실제 루프 연결이 약한 데이터
7) buildings.json / building_upgrades.json
- 위치: Assets/Resources/Cinderkeep/data
- 로드:
  GameDataManager.LoadBuildingData()
  GameDataManager.LoadBuildingUpgradeData()
- 다음 연결 후보:
  BuildingManager: 생성할 건축물 ID와 프리팹 선택
  BuildingHp: 건축물 체력
  BuildingSpot: 설치 가능 건축물 종류
  PlayerBuild: 건축 비용과 설치 요청
- 현재 기준:
  PlayerBuild는 BuildingManager를 통해 건축을 요청합니다.
  다만 건축 비용, 건축물 ID, 업그레이드는 아직 데이터 기반 체감이 약합니다.

8) crafting_recipes.json / crafting_stations.json
- 위치: Assets/Resources/Cinderkeep/data/crafting_recipes.json
- 위치: Assets/Resources/Cinderkeep/data/crafting_stations.json
- 로드:
  GameDataManager.LoadCraftingRecipeData()
  GameDataManager.LoadCraftingStationData()
- 다음 연결 후보:
  제작대, 용광로, 장비 제작 UI
  PlayerModel 자원 소비
  ToolData, WeaponData, ArmorData 해금
- 현재 기준:
  crafting_recipes.json은 무엇을 만들지 관리합니다.
  crafting_stations.json은 어떤 시설에서 제작/제련이 열리는지 관리합니다.
  제작대/용광로/모루는 Building 데이터와 CraftingRecipe 데이터를 함께 사용합니다.
  기본 비용은 제작대 Wood 10, 용광로 Stone 10, 모루 IronIngot 10입니다.
  CraftingUI는 제작 가능한 목록, 비용, 제작 가능/자원 부족 상태를 표시합니다.
  CraftingRecipeButtonView는 제작 목록의 한 줄 버튼입니다.
  CraftingRecipeExecutor는 비용 차감과 결과 지급만 담당합니다.

9) smelting_recipes.json
- 위치: Assets/Resources/Cinderkeep/data/smelting_recipes.json
- 로드: GameDataManager.LoadSmeltingRecipeData()
- 조회:
  GameDataManager.GetSmeltingRecipe()
  GameDataManager.SmeltingRecipeDataList
- 사용:
  FurnaceStation: 왼쪽 입력 자원, 진행률, 오른쪽 결과 주괴 수량
  FurnaceUI: 입력칸, 진행 게이지, 결과칸, 시작/회수 버튼 표시
- 조절 가능:
  _inputResourceId: 투입할 광석 ID
  _inputAmount: 한 번 제련에 필요한 광석 수
  _outputResourceId: 결과 주괴 ID
  _outputAmount: 결과 주괴 수
  _smeltSeconds: 제련 시간
  _requiredStationTier: 필요한 용광로 티어
- 현재 기준:
  IronOre -> IronIngot
  GoldOre -> GoldIngot
  AdamantiumOre -> AdamantiumIngot
  용광로 UI는 이 값을 읽어 왼쪽 입력칸, 진행률, 오른쪽 결과칸을 표시합니다.

10) armors.json / loot_drops.json / cinderheart_upgrades.json / status_effects.json / bosses.json / boss_patterns.json
- 위치: Assets/Resources/Cinderkeep/data
- 현재 기준:
  GameDataManager가 로드할 수 있는 준비 데이터입니다.
  아직 실제 플레이 체감과 강하게 연결된 상태는 아닙니다.
  아침 보상, 보스전, CinderHeart 성장, 장비 성장 작업에서 단계적으로 연결합니다.

JSON Edit Checklist
1) 바꿀 JSON 파일을 하나만 정합니다.
2) 바꿀 _id를 확인합니다.
3) 수치를 바꾼 이유를 짧게 적습니다.
4) 관련 스크립트를 확인합니다.
5) Unity에서 Play 후 Console 오류를 확인합니다.
6) 플레이 체감이 바뀌었는지 직접 확인합니다.
7) fallback 값으로도 안전하게 동작하는지 확인합니다.

UI Link Guide - 4.39
1) 제작 UI
- 스크립트: Assets/Scripts/UI/CraftingUI.cs
- 버튼 슬롯: Assets/Scripts/UI/CraftingRecipeButtonView.cs
- 제작 실행: Assets/Scripts/Systems/CraftingRecipeExecutor.cs
- 연결 위치:
  UIManager의 _craftingUI에 CraftingUI를 연결합니다.
  CraftingUI의 _recipeSlots에는 미리 만든 버튼 슬롯들을 배열로 연결합니다.
  버튼 슬롯마다 CraftingRecipeButtonView를 붙이고 TMP_Text, Button을 Inspector에서 연결합니다.
- 규칙:
  UI는 코드에서 새로 만들지 않습니다.
  제작 버튼 프리팹이나 씬 오브젝트를 미리 만든 뒤 연결합니다.

2) 용광로 UI
- 스크립트: Assets/Scripts/UI/FurnaceUI.cs
- 제련 실행: Assets/Scripts/Systems/FurnaceStation.cs
- 연결 위치:
  UIManager의 _furnaceUI에 FurnaceUI를 연결합니다.
  FurnaceUI에는 왼쪽 입력칸 TMP, 진행 Slider, 오른쪽 결과칸 TMP, 시작/회수 Button을 연결합니다.
- 현재 기준:
  드래그 앤 드롭 UI는 아직 2차 작업입니다.
  1차에서는 FurnaceUI.SetInputResource()로 선택된 광석 ID와 수량을 넘길 수 있게 준비했습니다.
  이후 DropSlot 컴포넌트를 만들면 왼쪽 칸에서 SetInputResource()만 호출하면 됩니다.

3) 상호작용 흐름
- 스크립트: Assets/Scripts/Systems/CraftingStationInteractable.cs
- 제작대:
  E 상호작용 -> CraftingStation.OpenStation() -> UIManager.OpenCraftingUI()
- 용광로:
  E 상호작용 -> CraftingStation.OpenStation() -> UIManager.OpenFurnaceUI()

Current 4.39 Note
- ResourceNode, PlayerToolController, PlayerToolUse, PlayerAttack, EnemySpawnPoint, GameFlowController는 실제 플레이 수치에 JSON을 사용합니다.
- buildings.json, building_upgrades.json, crafting_recipes.json, crafting_stations.json, smelting_recipes.json은 제작대/용광로/건축 비용 작업의 기준 데이터입니다.
- CraftingUI, FurnaceUI는 프리팹/씬에 준비된 UI를 표시하는 1차 뼈대입니다.
- 드래그 앤 드롭, 장비 인벤토리, 제작 결과 장착은 4.40 이후 PlayerInventory/EquipmentModel 작업과 함께 확장합니다.
- JSON 값이 없을 때는 각 컴포넌트의 Inspector fallback 값으로 안전하게 동작해야 합니다.
- 밸런스 수정은 한 번에 한 JSON, 한 ID, 한 목적만 바꾸는 방식이 안전합니다.
- 팀원이 브랜치를 올릴 때는 "어떤 JSON의 어떤 ID를 바꿨는지"와 "어떤 스크립트에서 체감되는지"를 같이 적어주세요.
