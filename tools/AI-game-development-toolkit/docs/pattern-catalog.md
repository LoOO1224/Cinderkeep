# 패턴 카탈로그

이 문서는 패턴 이름의 목록이 아니라 수집 시 사용할 체크리스트다. 각 패턴 문서는 이후 `문제`, `적용 조건`, `구조`, `Unity 구현`, `비용`, `대안`, `출처`를 갖춰야 한다.

## 게임플레이

| 패턴 | 해결하려는 문제 | 적합한 경우 | 주요 위험 |
|---|---|---|---|
| State | 상태별 동작과 전환 조건이 뒤섞임 | 캐릭터, UI, 게임 흐름의 상태가 명확함 | 상태 폭증, 전환 규칙 분산 |
| Strategy | 같은 목표의 알고리즘을 교체 | 타기팅, 이동, 점수 계산을 바꿔야 함 | 작은 분기를 불필요하게 객체화 |
| Command | 요청을 실행과 분리 | 입력 기록, undo, replay, 네트워크 입력 | 수명과 소유자가 불명확한 Command 누적 |
| Composite | 단일 효과와 효과 묶음을 동일 취급 | 스킬/퀘스트 조건의 조합 | 디버깅하기 어려운 깊은 트리 |
| Modifier/Decorator | 능력치를 조합해 확장 | 장비, 버프, 업그레이드 | 적용 순서와 중복 규칙 불명확 |
| Ability/Effect | 발동 규칙과 결과를 분리 | RPG 스킬, 카드 효과 | 만능 데이터 DSL로 비대해짐 |
| Blackboard | 여러 AI 노드가 문맥 공유 | Behavior Tree, Utility AI | 타입 안전성 저하, 숨은 의존성 |

## 객체 생명주기

| 패턴 | 해결하려는 문제 | 적합한 경우 | 주요 위험 |
|---|---|---|---|
| Factory | 생성 과정과 구체 타입 결합 | 데이터 기반 생성, 여러 생성 정책 | 단순 `Instantiate`를 과도하게 감쌈 |
| Object Pool | 반복 생성/파괴 비용 | 탄환, VFX, 짧은 수명 객체 | 반환 누락, 초기화되지 않은 상태 |
| Prototype | 원형 설정에서 복제 | 데이터 정의 기반 객체 | 공유 데이터와 런타임 상태 혼동 |
| Spawn/Despawn Service | 생성·제거 규칙 통합 | 네트워크 또는 씬 경계 객체 | 전역 서비스 로케이터화 |
| Addressables ownership | 비동기 자산 수명 관리 | 대형/원격/교체 가능 콘텐츠 | load/release 비대칭, 참조 계수 누수 |

## 시스템 연결

| 패턴 | 해결하려는 문제 | 적합한 경우 | 주요 위험 |
|---|---|---|---|
| Observer | 발행자가 구독자를 몰라도 됨 | 동적 UI와 로컬 도메인 이벤트 | 구독 해제 누락, 호출 순서 의존 |
| SO Event Channel | 씬/Prefab 참조 경계 통신 | 디자이너가 채널을 명시적으로 연결 | 호출 그래프가 Inspector에 숨음 |
| Mediator | 여러 객체 간 상호작용 집중 | 화면·전투 하위 시스템 조율 | God Object로 성장 |
| Message Queue | 이벤트를 지연·순서화 | 네트워크, 턴, 비동기 처리 | 즉시성 기대와 실제 처리 시점 불일치 |
| Dependency Injection | 생성과 의존성 제공 분리 | 테스트와 조립 경계가 중요 | 작은 프로젝트에 컨테이너 과잉 도입 |

`Event Bus`는 범위가 넓은 이름이다. 전역 만능 버스보다 도메인별 Observer나 명시적 채널을 우선한다.

## UI

| 패턴 | 해결하려는 문제 | 적합한 경우 | 주요 위험 |
|---|---|---|---|
| MVP | View와 상태/행동 분리 | MonoBehaviour View를 얇게 유지 | Presenter가 모든 로직을 흡수 |
| MVVM/View Binding | 상태와 표시 자동 동기화 | UI Toolkit 데이터 바인딩 | 숨은 갱신 비용과 디버깅 난도 |
| UI State Machine | 화면 상태 전환 관리 | 메뉴, HUD, 팝업 상태가 명확 | 게임 흐름 FSM과 역할 중복 |
| Navigation Stack | 뒤로 가기와 계층 이동 | 모바일/콘솔 메뉴 | 모달과 게임 상태의 불일치 |

## 데이터와 저장

- **ScriptableObject Definition + Runtime Instance:** 에셋에는 불변 정의를, 플레이 세션 상태에는 순수 C# 객체나 컴포넌트를 둔다.
- **Repository:** 저장 매체와 도메인 로직을 분리할 필요가 있을 때만 둔다.
- **Save DTO:** 런타임 객체 그래프를 그대로 직렬화하지 않고 저장 스키마로 변환한다.
- **Versioned Migration:** 저장 버전과 단계별 변환을 명시한다.
- **GUID Reference:** 에셋/아이템 참조가 이름이나 배열 인덱스 변화에 깨지지 않게 한다.
- **Data-driven Effect/Item/Card:** 정의 데이터와 실행 상태, 실행기를 분리한다.

## 게임 루프와 흐름

- **Update Manager:** 수천 개의 동일 갱신이 실제 병목일 때만 고려한다.
- **Fixed Tick:** 네트워크 예측이나 결정적 순서가 필요할 때 렌더 프레임과 시뮬레이션을 분리한다.
- **Turn Scheduler:** 턴 순서와 행동 실행을 분리한다.
- **Time Service:** 게임 시간, 실제 시간, 일시정지 영향을 명시한다.
- **Scene/Flow State Machine:** 부트, 메뉴, 로딩, 플레이, 결과 화면의 전환과 정리 순서를 관리한다.
- **Composition Root:** 앱 시작 지점 한 곳에서 장기 수명 서비스를 조립한다.

## 온라인

- Server Authority
- Client Prediction
- Snapshot Interpolation
- Lag Compensation
- Replication
- Network Command
- Connection State Machine

온라인 패턴은 개별 기법보다 책임 경계가 우선이다. 누가 입력을 만들고, 누가 판정하고, 무엇을 복제하며, 실패 시 누가 정리하는지를 먼저 표로 만든다.

## 패턴 문서 완료 조건

패턴 하나를 “수집 완료”로 표시하려면 다음을 모두 충족한다.

1. 실제 해결하려는 문제와 적용하지 말아야 할 경우가 적혀 있다.
2. 최소 한 개의 공식 또는 명시적 라이선스 출처가 있다.
3. Unity 생명주기, 직렬화, 씬/Prefab 경계에 미치는 영향이 설명되어 있다.
4. 테스트 방법과 대표 실패 모드가 있다.
5. 외부 코드를 포함했다면 원 저작권 고지와 라이선스를 보존했다.
