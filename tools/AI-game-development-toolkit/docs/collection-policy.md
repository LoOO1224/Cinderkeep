# 수집 및 라이선스 정책

## 기본 원칙

GitHub에서 읽을 수 있다는 사실은 복사·수정·재배포 권한을 뜻하지 않는다. 라이선스가 없으면 저작권자가 기본 권리를 보유한다고 보고 코드 복사를 금지한다.

외부 프로젝트는 기본적으로 링크와 분석만 저장한다. 코드를 가져와야 한다면 별도 디렉터리에 출처, 커밋, 라이선스, 수정 사항을 기록한다.

## 소스마다 기록할 항목

- 저장소와 문서 URL
- 조사한 브랜치와 가능하면 커밋 SHA
- 기준 Unity 버전과 주요 패키지 버전
- 마지막 의미 있는 릴리스/유지보수 시점
- 저장소 소스 라이선스
- 에셋, 폰트, 음원, 플러그인의 별도 라이선스
- Asset Store 에셋 포함 여부
- Git LFS 또는 대용량 다운로드 필요 여부
- 교육 샘플, 버티컬 슬라이스, 실제 배포 게임 중 무엇인지
- 코드 재사용 가능 여부와 필요한 고지
- 현대화가 필요한 API

## 가져오기 전 확인 절차

1. 루트 `LICENSE`, `README`, `Third Party Notices`를 읽는다.
2. 가져올 파일 머리말과 상위 폴더의 추가 라이선스를 확인한다.
3. 소스와 에셋 조건을 분리한다.
4. 정확한 커밋 SHA를 기록한다.
5. 원문을 복사할 이유가 없다면 패턴을 이해한 뒤 독립적인 최소 예제로 작성한다.
6. 라이선스가 충돌하거나 불명확하면 링크와 설명만 남긴다.

## 저장소 구조 제안

실제 예제를 추가할 때만 아래 구조를 만든다. 아직 코드가 없으므로 빈 폴더는 만들지 않는다.

```text
patterns/
  state/
    README.md
    unity-example/
  observer/
    README.md
    unity-example/
genres/
  action-rpg/
  fps/
sources/
  <project-name>.md
```

외부 저장소 전체를 이 저장소 안에 vendor하거나 서브모듈로 추가하는 작업은 별도 결정으로 남긴다. 대형 Unity 프로젝트와 Git LFS 저장소를 무분별하게 포함하면 저장소 크기와 라이선스 검토 비용이 급격히 증가한다.

## 분석 템플릿

```markdown
# 패턴/프로젝트 이름

## 해결하는 문제
## 적용 조건
## 구조와 책임
## Unity 특화 고려사항
## 실제 소스 위치
## 대안과 트레이드오프
## 실패 모드
## 테스트 방법
## 출처와 라이선스
```
